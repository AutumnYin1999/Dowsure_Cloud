## 0. 给下一个 AI 的 TL;DR（先看这段）

我们要把现有的「卖家对话版」`SellerChatPage.tsx` 升级成一个**统一的对话式 Agent 页面**：
- 用户进来 → Agent 用**任务语言**问"你最想搞定哪件事" → **分流到卖家线 / 服务商线**；
- 每条线问 3 个问题（对话式收集）→ 实时算出**缺钱/价值金额** → 给诊断 → 收尾 CTA；
- 顶部一条**"价值感状态栏"**随对话逐格填满（访客 → 金额 → 方案）；
- **核心目标**：在对话里把**豆服云**（服务商权益平台）和**豆分期 DPL**（先用后付）**咬合**起来——见 §3。

**80% 的代码可复用** `SellerChatPage.tsx`（已有 3 问对话、parse 兜底、透明 mock `diagnose()`、KPI/维度卡）。本轮主要是：①加身份分流 Q0 ②加服务商线 ③加顶部状态栏 ④用有背书的系数替换拍脑袋系数 ⑤卖家线收尾接豆分期预申请。

## 1. 背景与目标

- **豆服云**：服务商增长引擎（权益包：平台展示位 / AI 拓客 / 嵌入式金融 / 账期风控模型 / 豆分期服务 …）。当前要研究的主产品。
- **豆分期 DPL（= TermPay）**：跨境卖家"先用后付"信用支付工具，嵌在 Dowalet 钱包里。卖家付物流/仓储/广告账单时可延期 60/90/120 天，放款直接打到**服务商账户**（受托支付）。
- **终极目标（CEO 战略，非本轮范围）**：6 Agent + Agent OS（见 `Dowsure_团队任务方案_v1.2.md`）。本轮只做**一个对话页**，是这个愿景的最小可演示切片。
- **本轮目标**：做一个统一对话式 Agent 页，前端 mock 演示"对话收集信息 → 算钱 → 把豆服云与豆分期串起来"。

## 3. 已定的产品决策 —— 豆服云 × 豆分期怎么链接（本文件的灵魂）

**链接论点：豆分期是同一笔交易的两端，卖家线和服务商线在豆分期处闭环。**

```
        卖家线                              服务商线
  「账单压力大」诊断出账压 ¥X        「想给客户做账期 / 想获客」
          │                                   │
          ▼                                   ▼
  用豆分期延期支付这笔             开通豆服云「嵌入式金融端口 +
  物流/仓储账单 → 释放现金 ¥X       账期风控模型 + 豆分期服务」
          │                                   │
          └──────────► 豆分期放款打给服务商账户 ◄──────┘
             卖家：延期付款，现金不断流
             服务商：提前收款、零坏账、客户粘性↑
             平台：促成交易，赚价值
```

- **卖家侧落点**：对话诊断出"约 ¥X 现金被账期占用" → CTA「用豆分期延期这笔账单，预计释放现金 ¥X」→ TermPay 预申请（复用现有 TermPay 组件）。
- **服务商侧落点**：服务商被诊断出"客户账期坏账风险 / 想获客" → 推荐豆服云权益包，其中**豆分期 = 让服务商给自己的卖家客户提供账期**（自己提前收款、零坏账），**AI 拓客 = 给服务商精准卖家线索**。
- **闭环依据**：见 `豆分期产品介绍.pdf` 资金流图（Seller-Dowalet → Service Providers）。卖家用豆分期付的，正是服务商的账单。

> ⚠️ 这是基于已有材料的**最佳综合，待 product owner 最终确认**。若 owner 对链接逻辑有调整，先改本节，再改脚本。

## 5. 目标页面设计

### 5.1 路由
新建统一页 `AgentChatPage.tsx`，路由 `/agent`。**不动现有 `/seller/chat`、`/seller`、`/provider/*`**（与现有代码"对话版副本、原页面不动"的风格一致）。`GatewayPage` 可加入口指向 `/agent`。

### 5.2 三层结构（从上到下）
1. **价值感状态栏（新增，偷自 benchmark）**：一条 3~4 格的状态条，随对话填满。
   - 卖家线：`身份:卖家 → 账压金额:¥— → 可释放现金:¥— → 预申请:未发起`
   - 服务商线：`身份:服务商·物流 → 可拓客户:— → 推荐版本:— → 权益:未领取`
   - 初始全是"—/未评估"，每答一题点亮一格（参考 Payoneer/Wayflyer 的"看着自己变成有额度的人"）。
2. **对话卡片（复用现有）**：消息流 + 选项按钮 + 输入框 + 打字态。
3. **诊断/结果区（复用 + 扩展）**：KPI 金额卡 + 维度卡 + 收尾 CTA。

### 5.3 身份分流 Q0（新增，关键）
用**任务语言**问（JTBD，不要问"你的身份是？"）：

> 👋 我是豆服云助手。先帮你对上号——你现在最想搞定哪件事？
> - A 我在平台卖货，**钱卡在回款里 / 想找靠谱服务商** → 卖家线
> - B 我是**服务商，想获客 / 给客户做账期** → 服务商线

A→进卖家线 3 问；B→进服务商线 3 问。

## 6. 对话脚本（双线）

### 6.1 卖家线（复用现有 3 问，仅升级算钱 + 收尾）
| # | 问题 | 选项（已存在） | 作用 |
|---|---|---|---|
| 1 | 主要在哪个平台卖货？ | Amazon/Temu/TikTok Shop/SHEIN/eBay/独立站 | 路由 + 占压系数 |
| 2 | 每月 GMV 区间？ | 10万以下 / 10–50 / 80–120 / 120–300 / 300万以上 | 金额乘数 |
| 3 | 现在最头疼哪块？ | 成本太高/服务不稳定/响应太慢/不知道谁靠谱/账单压力大/担心合规 | 路由结局 |

**诊断输出（升级系数，见 §7）**：
- 现金占压 = 月GMV × 11~12%（账压金额）← 替换现有 `credit` 的 0.6 系数
- 缺货损失/季 = 月GMV × 7~14% × 断货占比 ← 替换现有 `stockout` 的 0.2
- 服务成本可优化 = 月GMV × (物流仓储占比 15~20%) × 可压缩比例

**收尾 CTA（链接豆分期）**：当 concern=账单压力大（或账压金额最大）→
> 「这笔现金缺口可以用**豆分期**延期支付物流/仓储账单填上，预计释放现金 **¥X**。」→ 按钮「生成豆分期预申请」→ 复用 `TermPayFlowCard`/`TermPayValueSection`（弹窗或下钻），状态栏"可释放现金/预申请"点亮。

### 6.2 服务商线（新增 3 问）
结局倒推：`获客线索` / `账期风控(豆分期)` / `品牌曝光`。

| # | 问题（任务语言） | 选项 | 作用 |
|---|---|---|---|
| 1 | 你是做哪类服务的？ | 物流/海外仓 · 整合营销 · 技术/工具(ERP/SaaS) · 其他 | 路由权益 + 算钱基准（七分类，见方案梳理结论四）|
| 2 | 业务体量大概？ | 按"年服务卖家数"或"年经手物流费/营收"分档 | 乘数 |
| 3 | 最想先解决？ | 多搞点客户 / 给客户做账期但怕坏账 / 想要品牌背书 | 路由结局 |

**诊断输出**：
- 获客价值：「豆沙包已触达近 10 万店铺、服务超 1 万卖家 → 按你的画像可推 **X 份**高价值卖家线索」（启航20/跃升80/领航120 份，见对服务商版 PDF）。
- 账期价值（豆分期）：「你年经手物流费约 ¥X → 用豆分期给客户做账期，可**提前回笼 ¥X、零坏账**，客户粘性↑」。
- 品牌价值：平台展示位 / Banner / 大卖有约等曝光量化。

**收尾 CTA（链接豆服云）**：按预算/最痛**推荐一个权益包版本**（必选 28,888 / 获客增长 / 增值服务 / 升级尊享），状态栏"推荐版本/权益"点亮。其中**豆分期服务 + 嵌入式金融端口 + 账期风控模型**作为"给你的客户做账期"的组合高亮——这就是与卖家线的咬合点。

## 7. 算钱系数表（带来源背书，写进一个 `coefficients` 常量）

| 系数 | 取值 | 来源（见方案梳理）|
|---|---|---|
| 现金占压率（卖家账压） | 月GMV × **11~12%** | JungleScout 痛点 + 卖家顾问截图实测 11.85% |
| 缺货损失率 | 销售额 × **7~14%** × 断货时长占比 | 亿邦：AWD 自动补货使缺货率↓15%、销售↑7-14%（反推）|
| 物流仓储成本占比 | 营收 × **15~20%** | 亿邦：商家财报抽样成本结构 |
| 豆分期/账期额度系数 | 月GMV/月流水 × **1.5**（封顶 $100万）| 豆分期 PDF；对标 Payoneer×1.4 / Wayflyer×1.5-3，落区间下沿=安全牌 |
| AI 拓客线索份数 | 启航 20 / 跃升 80 / 领航 120 份 | 对服务商版 PDF |
| 服务商获客难占比（话术用）| 43.48% 居首；67% 有融资需求 | 豆服云内部 24 份问卷 |

> 全部集中在一处，命名清晰（如 `const COEF = { cashLockRate: 0.115, stockoutRate: 0.10, ... }`），后续接后端只换这里。

## 8. 实施步骤（建议顺序）

1. **脚手架**：`AgentChatPage.tsx` + 路由 `/agent`（照 `App.tsx` 的 `STEP_PATHS`/`stepFromPath`/`navigate` 模式）。先把 `SellerChatPage` 整体拷过来跑通。
2. **抽象问题脚本**：把 `QUESTIONS` 从"写死卖家 3 问"改成 `{ seller: [...], provider: [...] }`，加 `Q0` 身份分流；`answerCurrent`/`stepIdx` 按"当前线"取脚本。
3. **顶部状态栏组件**：`<ValueBar line answers diag />`，根据已答字段逐格点亮。先卖家线，再服务商线。
4. **系数收口**：新建 `COEF` 常量，把 `diagnose()` 里 0.025/0.6/0.2 换成 §7 有背书的系数；金额文案带一句来源（"按行业均值约…"）。
5. **服务商线诊断**：写 `diagnoseProvider(answers)`，输出获客/账期/品牌三类价值 + 推荐版本（可借 `recommender.ts`）。
6. **卖家线豆分期落点**：收尾 CTA 接 `TermPayFlowCard`/`TermPayValueSection`（弹窗或 inline 下钻），状态栏点亮"可释放现金"。
7. **服务商线豆服云落点**：收尾推荐版本卡（可复用 `BenefitCard`/`EntitlementsMatrix`），高亮"豆分期+嵌入式金融"组合。
8. **自测 + 收尾**：两条线各跑一遍，金额联动正确、状态栏正确点亮、parse 兜底不崩。

## 10. 设计原则红线（不可让步）

1. **倒着设计**：每个问题必须"一题两用"（既分流、又是算钱输入）。不准加"纯属性、不算钱也不分流"的问题。
2. **价值感先行**：要手机号/预申请之前，必须先砸出最大的那个金额（foot-in-the-door）。
3. **豆分期 ×1.5 是安全牌**：对外口径锚定行业共识 1.4–3 倍，别吹大。
4. **领取 ≠ 申请**：额度/线索用"已为你预估 ¥X / X 份"的语气主动给（学 Payoneer），不要做成冷冰冰的申请表。
5. **金额全透明 mock**：系数集中、可追溯，严禁散落在 UI 里。
6. **任务语言开场**：问"想搞定哪件事"，不问"你的身份是"。
