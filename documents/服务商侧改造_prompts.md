## B 计划 · 权益自选工作台（待执行）

# 任务：把服务商推荐结果页改造为「可自选增减的权益工作台」（B 计划）

## 项目
Dowsure_Cloud（React 18 + Vite + TS + Tailwind，深色 Agent OS 风格，纯前端 mock）。
主要改动：src/components/RecommendationPage.tsx（推荐结果页）。
可新增：一个轻量的"选购状态"hook 或在页面内用 useState 管理。
不要改 recommender.ts 的 recommend() 入口签名；不要改问卷与诊断。

## 背景（现状）
- 主流程：问卷 → analyzing → diagnosis(诊断页) → result(推荐页)。
- 推荐页现在是「只读」：recommend(profile) 产出 RecommendationPlan，
  按类别分组展示 BenefitCard，有 KPI 卡（总价/付费项/赠送项）、QuickAdjust（调预算/目标重算）、
  完整权益明细（可折叠）、下一步建议、复制摘要。
- 痛点：服务商只能看推荐，不能自己加/减权益、不能看到"还差多少解锁尊享/独家"。
  用户 Note 明确要求：给详细信息 + 能看到所有权益 + 可自行增减 + 有下单入口。

## 目标：把结果页升级为「方案工作台」
保留 Agent 推荐作为默认选中集，但让服务商能在此基础上自由勾选增减，实时看到价格与解锁进度。

### 1. 选购状态
- 进入页面时，默认选中 = recommend(profile) 推荐出的 items（required/strong/optional 默认勾选；
  gift/unlock 作为"随单赠送/满额解锁"展示，不可手动取消）。
- 用一个 selectedIds: Set<string>（或等价结构）管理用户的勾选。
- 列出全部权益（来自 listAllBenefits() / KNOWLEDGE_BASE），不只是推荐项：
  - 推荐项：带"Agent 推荐"标记 + 推荐理由（复用现有 reasons / matchedRules 展示）。
  - 未推荐项：正常展示，可被用户主动勾选加入。
  - base 基础包：必选，不可取消。

### 2. 实时计费 + 解锁进度
- 顶部/侧边放一个"方案小计"区，随勾选实时更新：
  - 付费项小计（排除 gift/unlock）。
  - 满 20 万 → 解锁尊享、满 80 万 → 赠送独家：用进度条展示"当前 X 万 / 距下一档还差 Y 万"。
  - 达到阈值时，对应 unlock/gift 项自动点亮为"已解锁/已赠送"。
- 解锁逻辑请复用 schema 里的 BUDGET_FLOOR / 阈值常量与 recommender 的口径，保持一致。

### 3. 交互
- 每个权益卡片加一个加入/移除的勾选控件（checkbox 或 toggle）。
- 可按类别分组（沿用 CATEGORY_GROUP_ORDER 与 CATEGORY_META）。
- 提供"恢复 Agent 推荐"按钮：一键回到默认推荐集。
- 保留现有：KPI 概览、TermPay 价值模块、复制摘要、修改问卷、重新开始。
- 复制摘要 / 下一步建议要基于"用户当前实际选中集"，不再是固定 plan.items。

### 4. 下单入口（仅入口，闭环在 C 计划做）
- 底部主 CTA 改为"确认方案并下单 →"，点击先用 onConfirmOrder 回调（可暂时 showToast 占位）。
- 不要在本计划实现补充信息/确认单页/提交（那是 C 计划）。

## 约束
- 纯前端 mock；不改 recommend() 入口签名；解锁/价格口径与现有引擎一致。
- 保持深色 Agent OS 视觉风格与现有组件复用（Button、BenefitCard、KPI 卡、Toast）。
- 合规话术保留：Dowsure 提供技术/数据/风控/连接，TermPay 资金与最终授信由合作银行/资金方承担。

## 验收
- 进入推荐页：默认勾选 = Agent 推荐集，小计与现有总价一致。
- 勾选一个未推荐的付费权益 → 小计实时增加；勾到 ≥20 万 → 尊享项点亮为已解锁；≥80 万 → 独家项点亮为已赠送。

## Agent 2 · 账期「融资路径对比」内联模块（待执行）

# 任务：在卖家账期诊断结果中插入「三种筹资方式对比 → TermPay」（Agent 2）

## 项目
Dowsure_Cloud（React 18 + Vite + TS + Tailwind，深色 Agent OS 风格，纯前端 mock）。
改动文件：src/components/SellerHomePage.tsx（卖家智能服务台，账期测算在 computeTerm / 账期结果区）。
本轮只做前端展示，不接后端、不做真实申请。

## 背景与位置
卖家账期链路：账单诊断 → 判断现金是否够付 → （不够时）该怎么筹钱 → 落到 TermPay。
现状：computeTerm 算出压力等级（低/中/高/待确认）后，结果区直接给「TermPay 预申请」按钮，
中间缺少「为什么是 TermPay 而不是贷款/股权」的对比，故事是断的。

要在【账期判断结果卡之后、TermPay 预申请按钮之前】内联一个「三种筹资方式对比」模块。

## 触发规则（按压力等级分级）
- 压力「高」（现金不够）：完整展示三栏对比，金融科技栏高亮，底部接 TermPay 预申请。
- 压力「中」：展示轻量入口（一行「想了解其他筹资方式？」可展开看对比），不强推。
- 压力「低」：不展示（不缺钱，避免变推销）。
- 「待确认」：可复用「中」的轻量入口。

## 三栏内容（按用户既定口径，纯客观对比，不夸大）
1. 银行贷款：申请周期长、审核严格、需到期还本付息（可能需抵押/担保）。
2. 股权融资：资金确定性低、尽调资料复杂、周期久、稀释股权且需让渡部分经营参与。
3. 金融科技产品（TermPay，豆分期升级版）：申请简单、批复快、随借随还、按账期灵活延期/分期。
   → 自然作为最契合「大额账单+回款错配」场景的选项胜出，但不要用"最佳/保证/独家"等夸张词。

## UI 要求
- 内联在账期结果区，沿用现有 seller-desk.css 暗色卡片风格，三栏在桌面横排、移动端纵排。
- 金融科技栏用现有高亮/渐变描边样式区分，但保持克制。
- 模块底部承接现有「TermPay 预申请」CTA（openContact('TermPay 预申请')）。
- 加一行合规小字：额度/费率/期限以合作资金方审核与正式协议为准，Dowsure 不放款。

## 约束
- 纯前端 mock；复用现有 computeTerm 的等级结果，不改其计算口径。
- 不新增路由/页面（内联即可）；不引入新依赖。
- npm run typecheck 必须通过。

## 验收
- 账期压力「高」：结果卡下方出现三栏对比，金融科技栏高亮，可点 TermPay 预申请。
- 「中/待确认」：出现可展开的轻量入口，展开后见同一份对比。
- 「低」：不出现对比模块。
