## 1. 当前架构地图（拉到最新 `f0525fc` 之后）

入口是一个 **Gateway（`/`）**，分两侧人设：

### 🟪 服务商侧 `/provider/*`（“Agent 3：服务商 deal”）—— 已是完整闭环
`问卷 → analyzing → 经营诊断 → 权益自选工作台 → 下单 → 服务进度`

| 步骤 | 路由 | 组件 | 说明 |
|---|---|---|---|
| 问卷 | `/provider/intake` | `QuestionnairePage.tsx` | 服务商填经营信息，产出 `ProviderProfile` |
| 分析 | `/provider/analyzing` | `AnalyzingPage.tsx` | loading 动画 |
| 诊断 | `/provider/diagnosis` | `DiagnosisPage.tsx` + `core/diagnosis.ts` | 经营现状**定性**诊断（获客/账期/压力/现金流 4 维 + 交叉洞察 + 行动清单 + 痛点→方案） |
| 推荐 | `/provider/result` | `RecommendationPage.tsx` | 权益自选工作台（可勾选增减、实时计价、满额解锁） |

### 🟦 卖家侧 `/seller`（“Agent 1 找服务商” + “Agent 2 账期优化”）—— 全在一个大组件里
- `SellerHomePage.tsx`（约 1817 行，用 `seller-desk.css`）。
- 形态是**渐进式折叠长表单**（不是对话式）：选意图（找服务商 / 账期 / 都要）→ 填品类/阶段/平台/国家/服务/偏好拖拽/顾虑/账单明细 → **右侧内联输出**：
  - 3 个服务商推荐（`PROVIDERS` mock，带匹配理由 + 风险点 weakness）✅
  - 账期诊断（`computeTerm` → 压力分级 低/中/高/待确认）✅
  - 三种筹资方式对比（银行贷款 / 股权 / TermPay 金融科技，TermPay 高亮）✅
  - TermPay 预申请 CTA ✅

- `/seller/provider` → `ProviderProfilePage.tsx`（服务商详情页）。

## 2. 本次工作目标（卖家侧 · 对话式轻量入口 + 金额诊断）

来自 CEO 对卖家侧的反馈（原话要点）：
1. **先问 3 个问题就算痛点**：平台、月 GMV、当前需求/账期。
2. **输出一个金额结果**：“你这个阶段每月可能有 ¥X 服务成本浪费 / ¥Y 账期压力 / ¥Z 旺季缺货风险。”
3. **推荐 3 个服务商**，每个都要讲“为什么适合你” + “风险点”。
4. 涉及账期 → 直接生成 **TermPay 预申请**：“这张海外仓/物流账单可延期，预计释放现金流 ¥X。”
5. **CTA 改写**：`获取报价 + 账期方案` / `让顾问帮我对比 3 家` / `生成 TermPay 预申请`。

补充约束（用户口径）：本次只做一个 **case + 前端示例数据**即可，后续再替换后端；交互想做成**像聊天那样**——agent 一问一答、有选项可点、底部常驻一个**自输入框**，再配一个 **「填入示例」按钮逐题填**（demo 用，预置数据保证诊断结果漂亮）。自由输入没接 LLM，用**关键词映射 + 兜底**处理，接口留干净给后端。

## 3. 差距分析（CEO 5 点 vs 现状）—— 关键结论

| CEO 要的 | 现状 | 动作 |
|---|---|---|
| ① 只问 3 个问题（对话式） | ❌ 现有 `SellerHomePage` 是**超长折叠表单**，正是“问卷太多”问题本身 | **要新建**（对话式 3 问） |
| ② 金额诊断（亮 ¥ 数字） | ❌ 现有诊断是**定性**的（压力低/中/高；`core/diagnosis.ts` 也只给状态不给钱） | **要新建**（一小段透明 mock 系数算金额） |
| ③ 3 个服务商 + 理由 + 风险 | ✅ 已有（`SellerHomePage` 的 `PROVIDERS` + ProviderCard，含 reason/weakness） | **复用** |
| ④ 账期 → TermPay 预申请 | ✅ 已有（`computeTerm` + 筹资对比 + 预申请 CTA） | **复用** |
| ⑤ 新 CTA 文案 | 🟡 部分（发起需求/TermPay预申请/在线咨询） | 微调即可 |

**一句话：我们要做的“对话式 3 问 + 金额诊断”，正好补上现有重表单缺的 ①②，而 ③④⑤ 直接复用现成的。不是重复造轮子。**

## 4. 推荐方案（待用户最终拍板）

因为卖家侧逻辑现在都集中在 `SellerHomePage.tsx`，直接改它风险大。推荐先做**独立实验页**，不碰主组件：

- **新增路由**（如 `/seller/chat`）+ 新组件（如 `SellerChatPage.tsx`），挂进 `App.tsx` 的 `stepFromPath` / `STEP_PATHS`（注意：不要再用旧的 `startsWith` 土路由，要顺着新机制加）。
- **交互**：agent 一问一答（3 问：平台 / 月 GMV / 当前最痛）+ quick-reply 选项点选 + 底部常驻输入框 + 「填入示例」逐题填。
- **复用**：尽量复用 `SellerHomePage` 里的 `PROVIDERS` 数据与 `computeTerm` 账期算法（必要时抽到独立模块共享），以及 `seller-desk.css` 风格，保证视觉/数据一致。
- **新增**：金额诊断 headline（透明 mock 系数，单独一个函数/文件，便于后续替换后端）。
- 自由文本：关键词映射 → 枚举，命中不了就存 note + 走默认，演示不卡。

### 待拍板的三个方向
1. **独立实验页 `/seller/chat`**（推荐）：零冲突，演示“轻量版”，CEO 认可后再合并进 `/seller`。
2. **直接改造 `/seller` 入口**：在 `SellerHomePage` 前面加一层对话式精简问法——更彻底，但要重构大组件。
3. **只补“金额诊断”**：不做对话，先把“亮金额”加到现有流程，最小改动。

> 下一步：先和用户确认走 1/2/3，再细化具体实现方案（用户倾向先讨论方案、可能新开对话再写代码）。
