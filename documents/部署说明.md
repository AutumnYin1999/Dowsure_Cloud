# 部署说明 · 豆服云 Dowsure Cloud

> 纯前端站点（Vite 构建产物 `dist/`），部署 = 把 `dist/` 放到阿里云 ECS 上 nginx 指向的目录。
> 有两种方式：① GitHub Actions 自动；② 手动脚本。当前以**手动脚本**为主。

---

## 方式一：手动脚本（推荐 / 当前在用）

任何人或 agent 拿到仓库后，三步即可部署：

```bash
# 1. 复制配置模板并填写（仅需做一次）
cp scripts/deploy.env.example scripts/deploy.env
#    填好 ECS_HOST / ECS_PORT / ECS_USER / ECS_DEPLOY_PATH / ECS_SSH_KEY
#    （这几项与 GitHub Actions 的 5 个 Secret 完全一致）

# 2. 运行（Linux / Mac / WSL / Git-Bash）
bash scripts/deploy.sh
```

Windows 原生（PowerShell）：

```powershell
powershell -ExecutionPolicy Bypass -File scripts\deploy.ps1
```

脚本做的事：`npm ci → typecheck → build → scp dist/ 到服务器临时目录
→ 服务器端：备份当前站点 → rsync --delete 同步 → chown 回 www-data`。
（用 SSH 密钥免密登录；服务器端有 rsync，没有则退回 cp。）

**安全护栏**：脚本会拒绝把 `ECS_DEPLOY_PATH` 设成 `/`、`/var`、`/var/www` 等目录，避免误删。
**自动备份**：每次部署前把现有站点复制成 `<目录>.backup.<时间戳>`，出问题可回滚。

`scripts/deploy.env` 已被 `.gitignore` 忽略，**不会提交、不会泄密**。

## 🏠 新机器首次部署（如在家里接着改）

脚本（`scripts/deploy.sh` / `deploy.ps1`）会随 git 同步到新机器，但有**两样东西不在 git 里**（出于安全）：
1. **SSH 私钥** `~/.ssh/dowsure_ecs`
2. **`scripts/deploy.env`**（被 `.gitignore` 忽略）

新机器补齐这两样后，照样 `bash scripts/deploy.sh` 一键部署。两种补法：

### 方式甲：直接拷过去（最省事）
从已配好的机器，把这两个文件拷到新机器对应位置：
- `~/.ssh/dowsure_ecs`（私钥，注意权限 `chmod 600`）
- `scripts/deploy.env`（把里面 `ECS_SSH_KEY` 改成新机器上的私钥路径）

### 方式乙：让新机器上的 agent 重新配一把（用服务器 root 密码，一次）
> 适合手头没有私钥、但知道服务器密码时。需要本机有 Python + paramiko（没有就 `pip install paramiko`）。

1. 生成本机专用密钥：`ssh-keygen -t ed25519 -f ~/.ssh/dowsure_ecs -N ""`
2. 用密码把公钥装到服务器（把 `<公钥内容>` 换成 `~/.ssh/dowsure_ecs.pub` 的内容，密码用环境变量传、别写进文件）：
   ```python
   # ECS_PW=服务器密码 python 跑这段
   import os, paramiko
   c = paramiko.SSHClient(); c.set_missing_host_key_policy(paramiko.AutoAddPolicy())
   c.connect("8.163.94.219", 22, "root", os.environ["ECS_PW"], look_for_keys=False, allow_agent=False)
   pub = open(os.path.expanduser("~/.ssh/dowsure_ecs.pub")).read().strip()
   c.exec_command(f'mkdir -p ~/.ssh && grep -qF "{pub}" ~/.ssh/authorized_keys || echo "{pub}" >> ~/.ssh/authorized_keys; chmod 600 ~/.ssh/authorized_keys')
   ```
   （或者更简单：用阿里云控制台「终端连接」登录，手动把公钥追加进 `~/.ssh/authorized_keys`。）
3. 复制 `scripts/deploy.env.example` 为 `scripts/deploy.env`，按「生产参数表」填好（`ECS_SSH_KEY` 填本机私钥路径）。
4. `bash scripts/deploy.sh` 收工。

## 方式二：GitHub Actions 自动部署

`.github/workflows/deploy.yml`：**push 到 `main` 即自动**构建并 rsync 到 ECS。
依赖 5 个仓库 Secret（Settings → Secrets and variables → Actions）：
`ECS_SSH_KEY` · `ECS_HOST` · `ECS_PORT` · `ECS_USER` · `ECS_DEPLOY_PATH`

> 注意：这条流水线和「服务器上每分钟 git pull 的轮询脚本」是两回事。
> 轮询脚本在 ECS 上、已被手动关闭；本工作流仍会在 push 时触发，
> 如需彻底只走手动，可在该 workflow 注释掉 `on: push` 或在 Actions 页禁用它。

## ⚠️ nginx 必须配 SPA 回退

本站是前端路由（`/seller`、`/seller/chat`、`/provider/*` 等）。
nginx 站点配置里 `ECS_DEPLOY_PATH` 对应的 server 块必须有：

```nginx
location / {
    try_files $uri $uri/ /index.html;
}
```

否则**直接访问 / 刷新** `/seller/chat` 这类深链接会 404。
（现有 `/seller` 能正常刷新，说明这条多半已配好。）

## 部署后自检

脚本最后一步（5/5）已会**自动**在服务器上跑 `nginx -t` + curl 自检首页/子路由的 HTTP 状态码并打印 `<title>`，正常应看到 `home/首页: 200`、`spa/子路由: 200`。此外建议手动再确认：

1. 打开 `http(s)://<ECS_HOST>/` 首页正常。
2. 访问 `http(s)://<ECS_HOST>/seller/chat` 并**按 F5 刷新**，不 404 = SPA 回退 OK。
3. 走一遍 3 问对话 → 出智能诊断。
